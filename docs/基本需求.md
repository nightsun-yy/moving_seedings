# 前端总结与当前实现说明

## 1. 前端现在是做什么的

当前前端是一个基于 `React + TypeScript + Vite` 的单页控制台，用来演示“智慧农业 / 育苗搬运”的数字孪生界面。

它现在不是一个普通的信息展示页，而是一个带 3D 场景的操作面板，页面里同时包含：

- 顶部系统状态栏
- 左侧 MES 控制面板
- 中间 Three.js 3D 机械臂与育苗架场景
- 左下日志区
- 右下识别状态图例

这个页面的重点不是表单和列表，而是把“寻苗、识别、抓取、出库”的流程先用前端动画和模拟状态跑通。

## 2. 当前技术栈

前端目录在 `frontend/`，主要技术如下：

- `React 18`：负责页面组件和状态管理
- `TypeScript`：负责类型约束
- `Vite 5`：负责开发服务器和打包
- `Tailwind CSS 3`：负责页面样式
- `Three.js`：负责 3D 场景渲染
- `lucide-react`：负责图标

另外，`axios`、`react-router-dom`、`xlsx` 已经安装，但当前页面里还没有真正接入接口、路由或导出功能，属于预留依赖。

## 3. 主要文件作用

### `frontend/src/main.tsx`

这是前端入口文件，作用只有一个：把 `App` 挂载到浏览器页面里的 `#root` 节点上。

### `frontend/src/App.tsx`

这是当前前端最核心的文件，几乎所有页面逻辑都在这里，包括：

- 页面布局
- 系统状态展示
- 3D 场景初始化
- 机械臂动画
- 日志更新
- 模拟环境数据
- AI 巡检和出库流程

可以理解为：现在这个项目的前端还是“单文件主页面”阶段，功能已经成型，但还没有继续拆成多个组件。

### `frontend/src/index.css`

这里只负责引入 Tailwind 的三层样式：

- `@tailwind base`
- `@tailwind components`
- `@tailwind utilities`

### `frontend/vite.config.ts`

这里只启用了 React 的 Vite 插件，目前没有代理、别名、环境变量等更复杂的配置。

## 4. 页面里的核心状态

`App.tsx` 里主要维护了几组状态：

- `systemMode`
  - 控制系统当前处于 `AUTO`、`MANUAL` 还是 `SCANNING`
- `sysLog`
  - 维护左侧日志面板内容
- `envData`
  - 模拟温度、湿度、光照数据
- `armPos`
  - 记录机械臂的逻辑坐标
- `rackState`
  - 记录每个育苗仓位的状态

这些状态共同决定了页面上看到的数字、日志、按钮状态和 3D 动画效果。

## 5. 3D 场景现在做了什么

当前 Three.js 场景已经包含以下对象：

- 工作台
- 多层育苗架
- 托盘与苗盘状态球
- 三轴机械臂
- 末端执行器
- 摄像头
- 扫描锥光

前端在页面加载时会初始化场景、灯光、材质和模型，然后通过动画循环让机械臂向目标坐标缓动移动。

简单说，前端现在已经有了一个“能动起来的三维演示面板”，而不是静态图片。

## 6. 业务流程现在是怎么模拟的

页面里有两个比较关键的动作：

### `handleSmartPick`

这个函数负责模拟“一键 AI 寻苗与出库”：

1. 进入巡检状态
2. 按固定路径逐格扫描苗盘
3. 找到成熟苗时，记录日志
4. 模拟 Z 轴伸出和回收
5. 把该仓位状态改为空
6. 模拟搬运到出库口

### `handleReset`

这个函数负责模拟机械臂回原点，并记录日志。

## 7. 现在还没有真正接上的部分

虽然页面看起来已经像一个完整系统，但目前仍然是“前端模拟联调阶段”，下面这些部分还没有真正接到后端或 PLC：

- 没有真实 WebSocket 连接
- 没有真实 HTTP API 请求
- 没有真实 PLC 状态回传
- 没有真实摄像头识别结果

代码里目前通过 `sendCmdToPython()` 打印命令到控制台，表示后续应该在这里接 Python 网关或后端接口。

也就是说：

- 现在的页面是“前端仿真”
- 还不是“前后端联通后的生产版”

## 8. 这个前端目前的优点

- 页面效果已经比较完整，便于演示
- 3D 场景和业务流程已经能跑通
- 使用了 TypeScript，后续接接口更稳
- 使用 Vite，开发启动速度快
- 使用 Tailwind，后续调整界面效率高

## 9. 当前代码结构上的限制

现在的前端还有几个明显特点：

- `App.tsx` 过大，后续适合拆组件
- 3D 逻辑和页面 UI 写在一起，维护成本会逐渐上升
- 状态和常量还没有独立抽出
- 没有把“模拟数据”和“真实接口层”隔离
- 当前只有单页，没有真正用到路由

所以它适合作为第一版演示，但如果继续扩展，建议下一步做下面这些重构：

1. 把 `App.tsx` 拆成布局组件、控制面板组件、日志组件、3D 场景组件
2. 把 Three.js 初始化逻辑单独封装
3. 把 `sendCmdToPython()` 改成真正的接口层
4. 把仓位状态、日志状态和环境数据抽到独立模块
5. 根据后续页面数量决定是否启用路由

## 10. 一句话总结

当前前端已经不是空壳，而是一个“基于 React + Three.js 的智慧育苗数字孪生演示台”。

它现在最强的部分是：

- 3D 展示
- 状态联动
- 流程演示

它现在最需要补上的部分是：

- 真实接口
- 真实设备联动
- 组件拆分与工程化整理
